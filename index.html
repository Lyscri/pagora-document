01

Resumen Ejecutivo

Una pasarela de pagos (payment gateway) es el software que orquesta la autorización, captura y liquidación de dinero entre un comprador, un comercio, bancos y redes de tarjetas. Actúa como intermediario de confianza, cifrando datos sensibles, enrutando transacciones y manteniendo trazabilidad completa.

Decisión crítica #1: ¿Construyes tu propia infraestructura de procesamiento o integras un proveedor (Stripe, Braintree, Adyen)? La diferencia determina costos, tiempo, nivel de responsabilidad PCI y viabilidad legal. En el 80% de los casos, integrar es más inteligente que construir desde cero.
🏗️ Construir propia infraestructura

Control total, márgenes mejores a escala. Requiere licencias financieras, certificación PCI Level 1, equipo de +15 personas, 18-36 meses, inversión de $500K–$2M+. Apto para fintechs, bancos, grandes plataformas.

🔌 Integrar proveedor existente

Time-to-market de 2-8 semanas. Sin licencias de procesamiento. Costos por transacción (1.5%–2.9%). PCI SAQ A o A-EP. Ideal para e-commerce, SaaS, marketplaces, startups.

Decisión crítica #2: El país donde operas determina las normas aplicables: en EE.UU. rige el modelo Visa/MC directamente; en Europa aplica PSD2 + SCA obligatoria; en Latinoamérica cada país tiene su adquirente local y regulación propia (CNBV en México, BACEN en Brasil, CMF en Chile).
02

Conceptos Clave

Diferencias entre los actores del ecosistema

Gateway
La "tubería" técnica Software que recibe los datos del pago desde el frontend, los cifra, los envía al procesador y devuelve la respuesta. No mueve dinero. Ej.: Stripe, Braintree, PayU.
Procesador de pagos
El cerebro del enrutamiento Comunica a las redes de tarjetas (Visa/MC) con los bancos. Verifica fondos, gestiona autorización. Puede ser el mismo que el gateway o separado. Ej.: TSYS, First Data, Adyen.
Adquirente
El banco del comercio Institución financiera que tiene contrato con el comercio para aceptar pagos con tarjeta. Asume el riesgo de chargebacks. Ej.: Bancomer (México), Getnet (Brasil), BBVA Merchant.
Emisor
El banco del cliente Banco que emitió la tarjeta del comprador. Aprueba o rechaza la transacción verificando fondos, límites y reglas anti-fraude. Ej.: HSBC, Citibank, Nubank.
Merchant account
Cuenta de tránsito del comercio Cuenta especial (no bancaria estándar) donde se depositan los fondos autorizados antes de la liquidación. Separada de la cuenta corriente del negocio. Puede ser propia o usar la del proveedor (aggregator model).
Pasarela de pagos
El sistema completo Término paraguas que engloba gateway + procesamiento + UX de pago + gestión de estados + reportes + seguridad. Lo que construimos en esta guía.

Flujo de autorización simplificado

Cliente paga → Gateway cifra datos → Procesador envía a red (Visa/MC) → Red contacta Banco Emisor → Emisor aprueba/rechaza → Respuesta regresa por la misma cadena → Gateway notifica al comercio // Todo esto ocurre en < 2 segundos Liquidación (D+1 a D+3): Adquirente agrupa transacciones del día → envía a red → Red cobra/paga diferencias → fondos llegan a Merchant Account
03

Requisitos Técnicos

Backend

API REST/GraphQL

Endpoints para crear pagos, capturar, reembolsar, consultar estado. Versioned: /v1/payments. Stateless. Autenticación por API keys + JWT.

Motor de estados

Máquina de estados para cada transacción: created → authorized → captured → settled → refunded. Nunca se salta un estado.

Idempotencia

Cada petición de pago lleva un Idempotency-Key único. Si la misma key llega dos veces, se devuelve el resultado original sin reprocesar. Previene cobros duplicados por reintentos.

Reintentos con backoff

Para llamadas al procesador: exponential backoff con jitter. Máx. 3 intentos. Nunca reintentes pagos ya capturados. Circuit breaker para fallos masivos del proveedor.

Idempotencia — implementación

// Middleware de idempotencia (Node.js / Express) async function idempotencyMiddleware(req, res, next) { const key = req.headers['idempotency-key']; if (!key) return res.status(400).json({ error: 'Idempotency-Key required' }); const cached = await redis.get(`idem:${key}`); if (cached) return res.status(200).json(JSON.parse(cached)); // replay res.on('finish', async () => { if (res.statusCode < 500) { await redis.setex(`idem:${key}`, 86400, JSON.stringify(res.locals.body)); } }); next(); }

Webhooks

1
Recepción

Responder 200 OK inmediatamente. Nunca proceses síncronamente en el webhook handler. Encola el evento en Redis/SQS primero.

2
Verificación de firma

Validar HMAC-SHA256 del payload contra el secret compartido. Rechazar si difiere. Verificar timestamp (±5 minutos) para prevenir replay attacks.

3
Deduplicación

Guardar event_id en Redis/DB. Si ya existe, ignorar y responder 200. Los proveedores pueden enviar el mismo evento hasta 3 veces.

4
Procesamiento asíncrono

Worker consume la cola, actualiza estado en DB, dispara lógica de negocio (envío de email, activación de suscripción, etc.).

Conciliación

Proceso diario que compara lo que registró tu sistema vs. lo que reporta el proveedor/banco.

-- Query de conciliación diaria SELECT t.id, t.amount, t.status AS local_status, r.amount AS provider_amount, r.status AS provider_status, CASE WHEN t.amount != r.amount THEN 'AMOUNT_MISMATCH' WHEN t.status != r.status THEN 'STATUS_MISMATCH' WHEN r.id IS NULL THEN 'MISSING_IN_PROVIDER' ELSE 'OK' END AS reconciliation_status FROM transactions t LEFT JOIN provider_reports r ON t.provider_ref = r.id WHERE t.created_at::date = CURRENT_DATE - 1

05

Seguridad

PCI DSS — Lo más importante

Nunca almacenes datos de tarjeta en tu base de datos propia (PAN completo, CVV, PIN). Es ilegal y te expone a multas de $100K–$500K por evento de brecha. Usa tokenización desde el primer día.
SAQ A
El más simple — usa hosted fields o redirección Todo el pago ocurre en la infraestructura del proveedor. Tu servidor nunca toca datos de tarjeta. Self-assessment questionnaire de 22 preguntas.
SAQ A-EP
JavaScript propio en la página de pago Tu dominio sirve el JS que captura los datos, aunque los envíes directo al proveedor. 191 preguntas. Requiere penetration testing anual.
SAQ D / Level 1
Procesas o almacenas datos de tarjeta Auditoría anual por QSA certificado. 300+ controles. Para procesadores propios, más de 6M tx/año, o si te lo exige tu adquirente.

Cifrado y transmisión segura

🔒 TLS 1.3 obligatorio

Deshabilitar TLS 1.0 y 1.1. Usar HSTS con max-age=31536000. Certificate pinning en apps móviles. Renovación automática via Let's Encrypt o ACM.

🔑 Cifrado en reposo

AES-256 para campos sensibles (emails, nombres en logs). Gestión de claves con AWS KMS, HashiCorp Vault o Azure Key Vault. Rotación trimestral de claves.

🛡️ Autenticación

API Keys con prefijo + hash (nunca almacenar el key original). JWT para sesiones de usuario con expiración corta (15 min). Refresh tokens con rotación.

📋 Logs seguros

Nunca loguear PAN, CVV, contraseñas, tokens de sesión. Usar log scrubbing middleware. Logs inmutables en S3 con Object Lock. Retención mínima 12 meses.

Prevención de fraude

1
3D Secure 2.x (3DS2)

Autenticación adicional del portador de la tarjeta. Obligatorio en Europa (PSD2/SCA). Reduce chargebacks porque el emisor asume la responsabilidad. Implementar con stripe.confirmPayment() o equivalente.

2
Reglas de negocio propias

Velocidad por IP (máx. X intentos/hora), por tarjeta, por email. Bloqueo de países de alto riesgo. Umbral de monto sospechoso. BIN lookup para verificar país emisor vs. IP del comprador.

3
Motor de fraude externo

Kount, Sift, Signifyd o Stripe Radar. Machine learning sobre patrones de transacción. Score de riesgo por transacción. Configurar umbral de bloqueo y revisión manual.

4
Device fingerprinting

Recopilar device ID, user agent, IP, geolocalización, tiempo de sesión. Vincular dispositivo a historial de transacciones. Alertar si el mismo device usa múltiples tarjetas.

KYC / AML — Conoce a tu cliente

🪪 KYC Básico (onboarding)

Verificación de identidad: nombre, documento oficial, selfie + liveness check. Validación contra listas OFAC/SDN. Herramientas: Jumio, Onfido, Truora (LATAM).

🚨 AML — Monitoreo continuo

Perfiles de transacción por cliente. Detección de structuring (dividir grandes pagos). Reportes de operaciones sospechosas (SAR/ROS). Conservar registros 5+ años.

Riesgos legales críticos

Chargebacks excesivos: Si superas el 1% de chargebacks (Visa) o 0.9% (Mastercard), tu cuenta merchant puede ser suspendida o dada de alta en programas de monitoreo que incrementan costos significativamente.
Sin licencia de procesamiento: En la mayoría de países, procesar pagos de terceros sin una licencia de institución de pago es ilegal. Si solo cobras para tu propio negocio, el umbral es diferente.

Flujo Completo de una Compra

Flujo paso a paso: compra con tarjeta

1
Cliente llena el formulario de pago

El hosted field de Stripe/Braintree captura los datos de la tarjeta dentro de un iframe. El PAN nunca toca tu dominio. Tu JS obtiene un paymentMethodId.

2
Frontend → Tu API: /v1/payments

POST con { amount, currency, payment_method_id, idempotency_key, metadata }. Header: Idempotency-Key: uuid-v4. Tu backend valida y crea registro created en DB.

3
Motor anti-fraude

Score de riesgo en <200ms. Si score > umbral → declinar o solicitar 3DS. Verificar velocidad (intentos por IP, tarjeta, email en las últimas N horas).

4
Enviar al proveedor de pagos

Llamada a Stripe/Adyen con el token. Si requiere 3DS, devolver al cliente para autenticación. El proveedor enruta a través de Visa/MC al banco emisor.

5
Banco emisor responde

Aprobado: código de autorización. Rechazado: código de rechazo (fondos insuficientes, tarjeta bloqueada, etc.). Timeout: tratar como pendiente, no como error.

6
Actualizar estado en DB + responder al cliente

Estado → captured o failed. Registrar evento en transaction_events. Responder al frontend con el resultado. Si aprobado: disparar fulfillment.

7
Webhook del proveedor (confirmación asíncrona)

El proveedor envía payment.succeeded o payment.failed. Verificar firma. Deduplicar por event_id. Si el estado difiere del local → reconciliar.

8
Liquidación (D+1 / D+2)

El adquirente agrupa las transacciones del día y transfiere el neto (monto bruto - comisiones) a tu merchant account. Registrar en settlements.

Casos de Error y Manejo

APROBADO payment.succeeded

Estado → captured. Disparar fulfillment (envío, activar acceso, etc.). Enviar recibo al cliente.

→ Registrar evento. Actualizar inventario si aplica. Logging de éxito.

RECHAZADO insufficient_funds / card_declined / do_not_honor

Estado → failed. NO reintentar automáticamente. Devolver mensaje amigable al usuario (no el código técnico del banco).

→ Sugerir otro método de pago. Log del código de rechazo para análisis. NO exponer: "do_not_honor" → mostrar "Tarjeta no autorizada".

PENDIENTE payment.pending / processing

Estado → authorized o processing. Nunca entregar el producto aún. Esperar webhook de confirmación.

→ Mostrar "Pago en proceso". Job de polling cada 5 min para transacciones pendientes > 30 min. Timeout máximo: 24h.

TIMEOUT La llamada al proveedor excede el tiempo límite

Estado → pending. NUNCA marcar como fallido un timeout. El cobro pudo haberse realizado en el proveedor.

→ Consultar estado vía API con el provider_ref. Si no existe: crear nuevo pago con nueva idempotency_key. Si existe: sincronizar estado.

WEBHOOK DUPLICADO Mismo event_id recibido N veces

Los proveedores pueden re-enviar webhooks si no reciben 200 OK. Normal en alta disponibilidad.

→ Lookup en webhook_events por event_id. Si existe: responder 200 y no procesar. Si no: insertar y procesar. Operación debe ser idempotente.

FRAUDE DETECTADO Score alto / patrones anómalos

No revelar al cliente que fue detectado como fraude. Declinación genérica para no educar al atacante.

→ Bloquear temporalmente IP/email. Alertar equipo de riesgo. Conservar evidencia. Escalar a manual review si el score está en zona gris.

CHARGEBACK / CONTRACARGO Cliente disputó el cargo ante su banco

El proveedor te notifica via webhook. Tienes un plazo para presentar evidencia (7–20 días).

→ Congelar reembolsos sobre la transacción. Recopilar: comprobante de entrega, logs de acceso, comunicaciones con el cliente, IP del dispositivo. Responder dentro del plazo.
